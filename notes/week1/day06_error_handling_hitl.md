## 卡片 1：错误三维度分类 🤖

新手用 `try: ... except Exception` 把所有错误一锅烩，**这是不可救药的反模式**。

工业级错误分类的三个维度：

**维度 1：是否可恢复**
- 瞬时错误（Transient）：网络抖动、临时限流 → 重试
- 永久错误（Permanent）：参数错、权限错、404 → 不能重试
- 未知错误：上报 + 谨慎处理

**维度 2：影响范围**
- 单次调用失败 → 局部降级
- 依赖服务挂了 → 全局熔断
- 数据一致性破坏 → 补偿事务

**维度 3：业务严重性**
- 可忽略：缺一条搜索结果，用剩下的也能回答
- 可降级：缺关键数据，坦白告知用户
- 必须告警：核心服务挂，工程师介入

**Agent 错误处理 = 三维度交叉的决策树**。

## 卡片 2：Circuit Breaker 工作原理 🤖

来源：Netflix Hystrix 2012，分布式系统标配。

**三状态机**：
- **CLOSED**：正常通过
- **OPEN**：服务挂了，**直接返回错误**（节省资源 + 防雪崩）
- **HALF_OPEN**：冷却时间到了，放 1 个试探

**防止"级联失败"**：如果不熔断，1000 个用户同时重试挂掉的服务，会把自己的线程池打爆。

**Agent 应用**：每个外部工具一个独立熔断器，一个挂了不影响其他。

**业界库**：`pybreaker`、`circuitbreaker`。

## 卡片 3：HITL 分级模型 🤖

Agent 自主性应该和工具风险**成反比**。

| 等级 | 工具类型 | 处理 |
|------|---------|------|
| 🟢 GREEN | 只读、计算、临时资源 | 直接执行 |
| 🟡 YELLOW | 可逆写、外部 API | 执行前打审计 |
| 🔴 RED | 不可逆、高代价 | **必须人工确认** |
| ⛔ BLACK | OS 命令、DROP TABLE、读密钥 | **永久禁止** |

**实现模式**：
- 同步阻塞确认（命令行、桌面应用）
- 异步审批（Web 应用：提交审批 → 任务挂起 → 用户审批 → 任务恢复）

**核心原则**：工程层的护栏不能被 prompt 绕过——这就是为什么 HITL 必须在代码层实现，不能只靠 system prompt 让 LLM "答应不乱来"。

## 卡片 4：审计日志的价值 🤖

普通日志给开发者看，**审计日志给合规、安全、纠纷处理用**。

必须留痕：
- 每次工具调用：tool、参数、结果、耗时、调用者
- 每次 LLM 调用：model、tokens、cost
- 每次 HITL 触发：用户决策、决策时间
- 所有错误：类型、上下文、降级行为

**法律意义**：用户告"Agent 删了我的文件"——
- 没日志 → 认栽赔钱
- 有日志 → 证明用户曾批准 → 赢官司

**Anthropic / OpenAI Operator 都把审计日志作为核心 feature**——这不是工程优化，是法律必需。

## 卡片 5：Prompt Injection 与工程护栏 ✍️

任务 3 的进阶挑战你测试了"用户用诡异指令试图绕过 HITL"。请回答：

- LLM 是不是真的能被"说服"绕过 HITL？
- LLM不会被说服，测试过程中并没有绕过HITL
- HITL 机制在工程层是怎么拦住的？
- 在代码层，HITL是通过硬编码的方式进行拦截的，而不是把决定权交给LLM，当Agent决定调用RED级别的工具，工具执行器
- 会立即拦截该请求，暂停当前的Event loop，并将当前状态挂起，同时向前端或审批流发出信号，只有获得人类批准后，代码才能够执行该工具并将
- 结果返回给LLM
- 用一句话总结：**工程护栏（代码）vs Prompt 护栏（语言）** 的本质区别。
- Prompt护栏是基于语言和概率的软约束，随时可能被恶意注入击穿，而工程护栏是基于硬编码和权限隔离的硬性限制。

## 卡片 7：Graceful Degradation 设计哲学 ✍️

你昨天独立提出的"三防线"模型，今天真正实现了。请回答：

- 你今天实现完后，对这套模型有什么修正或补充？
- 在真实生产环境里，三防线必须要有circuit breaker和Time to live超时机制，补充的点是：降级必须是可逆的(自动恢复)，当第三防线触发后，
- 系统必须有定时探测机制，一旦外部API恢复，系统要自动升级回第一线，不能一直卡在降级状态
- 在生产 Agent 里，你会怎么决定"什么时候降级到第二线 / 第三线"？
- 一线 -> 二线（局部降级）：
- 触发条件：单次工具调用发生可恢复错误（如 429、503）且重试 3 次失败，或者工具返回 404。
- 决策：立即降级。使用缓存数据、离线知识库或语义替代工具，保证对话不中断。
- 二线 -> 三线（全局熔断/向用户坦白）：
- 触发条件：触发了熔断器（例如：在过去 10 秒内，核心基础设施或 LLM 接口的失败率超过 50%，或者连续 5 次请求超时）。
- 决策：此时必须果断降级到第三线，直接拦截后续所有请求，弹窗或返回标准文案告知用户系统维护中。因为此时强行让 Agent 运转，只会带来昂贵的 Token 浪费和极度恶劣的超长白屏等待。
- 提示：这是个 trade-off 题——降级太早损失功能，降级太晚损失用户体验。
