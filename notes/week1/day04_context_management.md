## 卡片 1：BPE Tokenizer 工作原理 🤖

BPE（Byte-Pair Encoding）通过统计语料中**最频繁的相邻字符对**，迭代合并形成 token 词表。

工程后果：
- 英文常见词组合频繁 → 一个 token 能装下几个字符（效率高）
- 中文每个字独立罕见组合 → 通常 1 字 = 1-2 tokens（效率低）
- 同样语义，中文 prompt 成本约为英文的 **3 倍**

实战建议：
- 跨语言对比时，token 数比字符数更能反映成本
- 选模型时不只看 context window，也要看 tokenizer 对你语言的友好度
- 最新中文模型（DeepSeek-V3、Qwen3）做了 tokenizer 优化，中文效率好得多

## 卡片 2：Lost in the Middle 现象 🤖

斯坦福 2023 论文发现：LLM 对长上下文的注意力呈 **U 形分布**——开头和结尾的信息找得准，中间的容易被忽略。

工程后果：
1. 不要无脑塞满 context window
2. 关键信息要么放最前（system prompt），要么放最后（最近的 user message）
3. 中间部分是上下文压缩算法的重点目标
4. "支持 200k context" 是物理上限，**有效注意力上限远小于此**

## 卡片 3：三种上下文管理策略对比 🤖

| 策略 | 实现 | 优点 | 缺点 | 适合场景 |
|------|------|------|------|---------|
| Sliding Window | 保留最近 N 条 | 简单、可控、零额外成本 | 老信息突然丢失 | 客服、单任务 |
| Summarization | LLM 把老消息摘要 | 保留早期信息 | 多 1 次 LLM 调用、有损 | 长对话陪伴 |
| RAG Memory | 历史进向量库 | 记忆容量无限 | 复杂、丢时序 | 长期助理、跨会话 |

**行业现状**：生产 Agent 90% 用滑窗 + 关键信息 pinning。MemGPT 类方案学术热但工业落地少。

## 卡片 4：messages 数组的 token 计算细节 🤖

OpenAI 的 messages token 计算 ≠ 简单把所有 content 的 token 加起来。每条消息还有**固定开销**：

- 每条消息约 **4 tokens** 用于 role 标识、分隔符
- 每个 response 还有 3 tokens 的固定开销
- 包含 `tool_calls` 的消息要把整个 JSON 序列化算 token
- `name` 字段额外 1 token

**实战意义**：用 `tiktoken` 精确算可以**预先**做决策（要不要压缩、要不要降级模型），而不是事后看 `usage` 发现已经烧完了。

## 卡片 5：策略选型的决策框架 ✍️

请基于你今天的实战体验回答：

- 给定一个新的 Agent 项目，你会用什么标准判断该选哪种上下文管理策略？至少列 3 个考量维度。
- 1.记忆的时序依赖度：任务是否依赖最近几轮的文本、语气、代码上下文，此维度我推荐Sliding Window
- 2.核心事实的跨度： 关键信息是否在数轮后仍需要调回？ 推荐Sliding Window + System Pinning
- 3.成本与延迟：首字延迟TTFT是否敏感？预算是否有限？敏感则用Sliding Window
- 提示：可以从"对话长度分布"、"是否需要跨轮事实记忆"、"成本预算"、"延迟要求"等角度思考。

## 卡片 6：摘要式压缩的潜在风险 ✍️

任务 3 的进阶挑战如果你做了，应该感受到摘要模式的"威力"也有"危险"。请回答：

- 摘要式压缩有哪些"看起来 work，实际有坑"的失效模式？至少列 3 种。
- 摘要式压缩发生错误时会发生幻觉滚雪球，错误会被无限放大，Agent彻底被带偏
- 对于细节把握不够精准，摘要会把可能看似无用的细节删掉，但用户后续问起，Agent却回答不出
- Prompt被注入与污染：如果历史对话中包含恶意文本，可能会在摘要中变成全局指令，导致Agent被成功注入
- 提示：想想"摘要错了怎么办"、"摘要被反复压缩怎么办"、"摘要里的细节优先级谁来定"。
- 在生产环境里，你会怎么缓解这些风险？
- 不只存摘要，保留一个原始消息的超轻量文本滑窗，同时也可以维护一个结构化的事实表格

## 卡片 7：上下文管理 vs 长上下文模型 ✍️

2025 年 Gemini 2.0 Pro 已经支持 2M 上下文，Claude 也在涨。请回答：

- 当上下文窗口足够大时，我们还需要上下文管理吗？为什么？
- 仍然需要，上下文关乎首字延迟TTFT和成本，即使模型的支持越来越长的上下文，但是将token塞进context内时，模型为了理解需要海量的Attention计算
- 所以首字延迟需要的时间会越来越长，随之成本也会负担不起，越发长篇的上下文塞入模型，每次请求都会将可能包含敏感隐私的历史数据在网络和服务器见传输，会增加数据泄露的风险
- 提示：从成本、延迟、Lost in the Middle、用户隐私等角度想
- 你的结论是"工程问题终将被算力解决"还是"工程优化永远必要"？为什么？
- 结论：工程优化永远是必要的，算力无法终结工程，只会改变工程的边界
