## 🎯 核心认知收获

- 认识到了 ReACT 范式下的agent并不是自己想干什么就能干什么
- 它也只是大模型和工程完美结合的产物，一切都是人类通过类似于“搭积木”的方式实现起来的

### 1. Agent 的"自主性"的本质

-现在我定义的“Agent 自主”，是在开发者划定的护栏内，模型通过一个确定性的程序循环，获得了一种在单步执行上的选择权：
-物理实现：就是 LLM 通过生成 finish_reason 来“投票”决定是调用工具还是停止。
-本质：它是一种通过精巧的循环结构和工程约束，将 LLM 在文本空间内的预测能力，
-转化为在预设动作空间内的、带反馈的连续决策过程。它不是魔法，而是上下文管理和状态机驱动的工程结果。

### 2. messages 数组是 Agent 工程的核心数据结构
- 是的，messages数组是LLM真正“看”到的东西，我们大部分的工作都是围绕messages数组来展开的


### 3. 长寿命 Agent 不是"记性好的作家"，是"精明的记忆管理者"
- 随着记忆越来越长，也就是messages数组里面的内容越来越多，为了让Agent仍然能够有效的帮助我们解决问题，同时控制随之而来的token成本问题，
- 所以我们要对记忆进行精明管理，通过过滤、压缩等手段，把真正的信息“记住”

### 4. 真实工具引入了 Agent 的"物理感"
- 真实工具能够让我们真切感受到当今的互联网生态，能够进行真正的行为交互，通过Agent帮我们实现这真切的物理感受。

### 5. 工程护栏 vs Prompt 护栏的本质区别
- 工程护栏永远是最可靠的护城河，它能够通过我们自己设下的硬性限制保护系统程序不出问题，Prompt归根结底是“喂”给LLM的内容，LLM会遵循Prompt生成相应的答案
- 如此将选择权交给LLM时，这种护栏就会有出现“护不住”的可能性
