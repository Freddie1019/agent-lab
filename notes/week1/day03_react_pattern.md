# Day 3 概念卡片：ReAct 循环

---

## 卡片 1：ReAct 是什么 🤖

**ReAct = Reasoning + Acting**，2022 年 Google 论文提出的 Agent 工作范式。

核心思想：让 LLM 在"思考 → 行动 → 观察"之间反复循环，直到任务完成。
Thought（思考）→ Action（调工具）→ Observation（看结果）
↑                                      ↓
└──────────────────────────────────────┘
（根据观察决定下一步思考）

**意义**：所有现代 Agent 框架（LangGraph、AutoGPT、Manus、OpenAI Assistants）的内核都是 ReAct 的变种。

---

## 卡片 2：ReAct 的代码本质 🤖

去掉所有花架子，ReAct 就是一个 `while 循环` + `finish_reason 判断`：

```python
while step < max_steps:
    response = llm.call(messages, tools)
    messages.append(response.message)

    if response.finish_reason == "stop":
        return response.message.content          # ← 任务完成
    if response.finish_reason == "tool_calls":
        for tc in response.tool_calls:
            result = execute_tool(tc)
            messages.append({"role": "tool", ...})
        # 继续循环
```

**关键认知**：所谓"Agent 自主决策"，物理实现就是 LLM 通过 `finish_reason` 来"投票"决定循环是否继续。

## 卡片 3：Native Tool Calling vs Prompt-based ReAct 🤖

| 维度 | Native Tool Calling | Prompt-based ReAct |
|------|---------------------|---------------------|
| 实现方式 | 用 LLM 厂商的 `tools` 参数 | 完全靠 prompt + 文本解析 |
| Thought 可见性 | 默认不可见（可通过 prompt 引导显式输出）| 天然可见 |
| 代码复杂度 | 简单（解析 JSON 结构） | 复杂（解析文本，易出错） |
| 模型要求 | 需要支持 Tool Use 的模型 | 任何模型都能用 |
| 生产环境 | ★ 主流选择 | 已基本被淘汰 |

**今天用的就是 Native Tool Calling + Prompt 引导显式 Thought 的混合方案**。

---

## 卡片 4：Agent 必须的护栏 🤖

任何能自主循环的 Agent，**必须**有这些护栏：

| 护栏 | 防止什么 | 实现方式 |
|------|---------|----------|
| **max_steps** | 死循环 | `for step in range(max_steps)` |
| **token 预算** | 成本失控 | 累计 `usage.total_tokens`，超限熔断 |
| **超时** | 单步卡死 | 单次 LLM 调用设 timeout |
| **错误捕获** | 工具异常崩溃 | 把错误作为 Observation 喂回 LLM |
| **运行报告** | 出问题查不到 | 记录步数/tokens/耗时/状态 |

**记住：没有护栏的 Agent 是定时炸弹。**

---

## 卡片 5：finish_reason 是 Agent 状态机的核心信号 🤖

Day 2 我们说 `finish_reason` 是"方向盘"，今天看到了具体怎么用：

| 值 | Agent 该做什么 |
|----|---------------|
| `stop` | 退出循环，返回最终答案 |
| `tool_calls` | 执行工具 → 结果塞回 messages → 继续循环 |
| `length` | 报警（上下文太长，需要压缩或放宽 max_tokens） |
| `content_filter` | 安全审查触发，返回降级回答 |

**ReAct 循环本质就是基于 finish_reason 的状态机**。

## 卡片 6：Thought 真的是"思考"吗？ ✍️

请基于今天任务 3 的体验思考并回答：

- 我们在 system prompt 里写了"先输出思考再调用工具"，LLM 输出的那段"思考"，它是真的在"思考"吗？
不是。从机制上讲，这仍然是语言模型根据 prompt 指令和上下文，通过预测下一个最可能的 token 生成的一段文本。我们给它定义了一个角色叫“Thought”，但它没有主观意识，也不是独立的认知过程。
它更像是一个“有声思维”的脚本，通过将推理过程文本化，来提高后续动作（工具调用或最终回答）的准确性。
- 这段"思考"对最终的工具选择有没有实际影响？还是只是给我们看的"表演"？
有，而且至关重要。 虽然它不是真正的思考，但这段生成的文本会成为上下文历史的一部分，直接影响模型在下一步生成 tool_calls 时的决策。
这就是“思维链”提示的作用：通过强制输出中间推理步骤，模型能处理更复杂的逻辑，减少幻觉，工具选择和参数填充会更准确。它不只是表演，它是模型利用自身生成能力来增强下一步决策的工程技巧。
- 提示：可以思考 OpenAI o1 / DeepSeek R1 的"展示推理过程"，和我们今天做的有什么区别？
这是根本性的不同。
我们做的：是应用层提示工程。推理过程只是模型最终为生成工具调用而“顺便”输出的普通文本，本质和写小说一样。
o1/R1 做的：是模型内部强化学习。它们用强化学习训练，学会在“内部思维空间”进行长时间、多路径、带反思的推理。我们看到的总结只是这个内部过程的冰山一角。
它们的“思考”是在私有、不对外暴露的上下文中完成的，是真正的“内部独白”，而不是应用层的 prompt 表演。

## 卡片 7：你眼中的 Agent 框架 ✍️

经过 Day 1-3 的手写实践，请你回答：

- 如果让你用一句话向新人解释"LangChain / LangGraph 这种 Agent 框架到底在做什么"，你会怎么说？
“它们提供了一套标准化的工程组件（如状态管理、循环控制、持久化），让你能像搭乐高一样，快速构建、调试和部署那些需要多步推理和工具使用的 AI 应用，而不必每次都从 while 循环写起。
- 这些框架相对于"手写 ReAct"，多了哪些价值？（至少说 2 点）
复杂状态与流程管理：手写循环只能处理简单的线性流程。框架内建了图、树、子图等结构，能轻松实现复杂的条件分支、并行执行和人在回路，这是手写循环难以维护的。
生产环境必需的非功能特性：框架开箱即用地解决了可观测性（如 LangSmith 追踪每一步的输入输出）、持久化（对话中断后恢复）、错误恢复（节点级重试）和流式处理等生产级问题，手写需要巨大的投入。
- 什么场景下你会选择"手写 ReAct"而不是用框架？
极速原型验证：只想快速测试一个想法或新的提示词，一个几十行的脚本远快于搭建框架环境。
极致轻量与低依赖：在一个资源受限的环境或一个只做单一、固定流程的微服务中，引入一个庞大的框架是过度设计。
深度定制化教学：为了完全理解 Agent 的核心原理，手写是必经之路。

## 卡片 8：今天最关键的工程认知 ✍️

用 2-3 句话总结：
- 你以前对 Agent 的"自主"有什么想象？
我以前可能想象 AI Agent 拥有某种神秘的、接近人类的自发意愿和长远规划能力，像一个黑箱一样“自己决定干点什么”。
- 现在写完 ReAct 循环，你怎么定义"Agent 的自主"？
现在我定义的“Agent 自主”，是在开发者划定的护栏内，模型通过一个确定性的程序循环，获得了一种在单步执行上的选择权：
物理实现：就是 LLM 通过生成 finish_reason 来“投票”决定是调用工具还是停止。
本质：它是一种通过精巧的循环结构和工程约束，将 LLM 在文本空间内的预测能力，转化为在预设动作空间内的、带反馈的连续决策过程。它不是魔法，而是上下文管理和状态机驱动的工程结果。
