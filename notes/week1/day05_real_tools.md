## 卡片 1：Search API 生态的演化 🤖

**2023 年以前**：搜索 API（Google CSE / Bing API）是给搜索引擎竞争者用的，**不是为 Agent 设计**——返回原始 HTML、SEO 排名结果、广告位。

**2023 年后**：出现了专为 LLM Agent / RAG 设计的新一代搜索 API：

- **Tavily** — 返回已清洗摘要 + 可选直接 Answer，专为 Agent
- **Exa** — 基于 embedding 的"语义搜索"，找"含义相近"的内容
- **Jina Reader** — Web → 干净 Markdown，专门解决"网页太脏"的问题
- **Firecrawl** — Agent 友好的爬虫，自动处理 JS 渲染、反爬
- **Perplexity Sonar** — 搜索 + LLM 总结一体化

**判断一个 API 是不是"Agent 友好"的标准**：
返回的格式是 LLM 可以**直接吃**的（Markdown / 清洗后纯文本 / 结构化 JSON），而不是给人看的 HTML 渲染。

## 卡片 2：网页抓取的真实复杂度 🤖

"抓网页"在 2025 年绝不是 `requests.get(url).text` 这么简单。常见问题：

- **超时**（服务器慢、网络抖动）→ `httpx` 的 `timeout` 参数
- **反爬（403 / 429）** → 加 User-Agent、降低频率、用代理
- **编码乱码**（GBK / UTF-8 / ISO 混杂）→ `chardet` 自动检测 / `apparent_encoding`
- **JavaScript 渲染**（SPA 网站）→ `Playwright` / `Selenium`
- **Cloudflare 反爬** → `cloudscraper` / 商业服务
- **HTML 噪音**（导航、广告、版权声明）→ `trafilatura` / `readability-lxml`
- **非文本内容**（PDF / 视频 / 图片）→ 类型检测 + 跳过 + 专用解析器

**核心原则**：Agent 工程师不该重新发明轮子做爬虫。爬虫是一个有 20 年积累的独立工程领域，应该用专业工具（Tavily / Firecrawl / Jina Reader）。

**懂得用什么、不自己写什么，是高级工程师的素养。**

## 卡片 3：工具设计的三个层次 🤖

### 层次 1：原子工具（Atomic Tools）

做一件最小的事：
- `search(query)` — 只搜索，不抓
- `fetch_url(url)` — 只抓一个 URL
- `extract_text(html)` — 只提取文本

**优势**：清晰、可组合、好测试
**劣势**：LLM 要自己编排多步，容易出错、慢

### 层次 2：组合工具（Composed Tools）

把几个原子组合成一个"业务动作"：
- `search_and_summarize(query)` — 搜索 + 抓 + 总结
- `compare_products(name_a, name_b)` — 分别查两个 + 对比

**优势**：LLM 调用次数少、效率高
**劣势**：灵活性差，LLM 不能微调中间步骤

### 层次 3：智能工具（Smart Tools）

工具内部自己用 LLM，等于"Agent 套娃"：
- `research(question)` — 自己规划 → 自己搜 → 自己总结 → 自己写报告
- `code_review(repo_url)` — 自己拉代码 → 自己分析 → 自己出报告

**优势**：主 Agent 调用一次就拿到完整结果
**劣势**：黑盒、难调试、成本叠加

### 选型决策

- 通用对话 Agent → 原子工具（LLM 灵活组合）
- 客服 Agent（流程固定）→ 组合工具（减少 LLM 思考步数）
- 复杂研究 Agent → 智能工具（让"研究"成为黑盒能力）
- 多 Agent 系统 → 混合（主 Agent 用智能，子 Agent 用原子）

**没有最优层次，只有最适合场景的层次。**

## 卡片 4：超时 / 重试 / 速率限制 🤖

所有外部调用（API / DB / Redis / 第三方服务）都必须有这三件套：

### 超时（Timeout）
- 作用：防止程序永久卡死等待
- 关键点：必须显式设置，不能依赖默认值
- 实战：`httpx.get(url, timeout=10.0)`

### 重试（Retry）
- 作用：处理瞬时网络故障
- 关键点：**区分"该重试 vs 不该重试"的错误**
  - 该重试：429（限流）、500/502/503（服务端）、网络错误、超时
  - 不该重试：400（参数错）、401（认证错）、403（权限错）、404（不存在）
- 策略：指数退避（1s → 2s → 4s → 8s）
- 实战：`tenacity` 库

### 限流（Rate Limiting）
- 作用：防止 bug 或恶意调用把额度刷光
- 关键点：客户端也要限流，不要只依赖服务端
- 实战：`asyncio.Semaphore`（并发限制）/ token bucket（QPS 限制）/ Redis 计数器（分布式）

### 记忆口诀

**永远假设外部服务会挂、会慢、会被滥用。**

## 卡片 5：工具结果体积控制策略对比 ✍️

请你结合今天的实战回答：

- 你今天用了"截断（Truncation）"策略来控制工具结果体积。这种策略有什么**致命缺点**？请举一个具体场景说明。
- 截断的致命缺点是 "信息首尾分布不对称"导致核心信息丢失，很多长篇分析文章的精彩论点和结论都在文章中后部，盲目按照字符数字截断
- 会导致核心答案的丢失；

- 提示：想想"信息密度不均"的网页——比如新闻列表页 vs 长篇分析文章
- 如果让你给一个研究型 Agent 设计"结果体积控制策略"，你会怎么组合 Truncation / Summarization / Chunking 三种方案？
- 我的想法是不只用单一策略，在生产环境中建立一个 “清洗->切片->智能检索/压缩”的管道
- 第一步先清洗，把拥有复杂信息的HTML清洗成干净的Markdown，第二步做语义切片，将清洗后的文本切成小块，用用户查询去小块里做语义检索
- 挑出相关度高的3-5个切片，第三步做摘要，如果挑出的切片依然大，可以用个廉价的小模型做单片摘要，最后把摘要拼接给主Agent

## 卡片 7：限流熔断时 Agent 的"降级行为" ✍️

任务 5 进阶挑战你触发了限流。请回答：

- 当限流命中时，你的 Agent 怎么处理的？它是诚实地承认"信息不全"，还是开始编造？
- 我测试过程中 Agent 的回复是：目前没有找到关于 xAI 2025 年旗舰模型的具体信息，还算承认信息不全的事实
- 你觉得**理想的 Agent 降级策略**应该是什么？（提示：考虑用户体验 + 信息可靠性的平衡）
- 理想的降级策略
- 防御第一线：工程层自动容错：利用 tenacity 在工具内部实现指数退避重试。如果遇到 429 且 Response 包含 Retry-After，代码应该让线程自动 sleep 对应时间再试，这一步对 LLM 是隐蔽的。
- 防御第二线：优雅的错误语义化（对 LLM 诚实）：如果重试依然失败，工具不要抛出代码异常，而是返回一句充满业务语义的话给 Agent：“错误：由于第三方搜索服务当前限流，无法获取最新数据。请基于你现有的知识库回答，并在回答中明确提示用户由于网络原因未能获取实时信息。”
- 防御第三线：主 Agent 行为降级（向用户坦白）：Agent 读取上述语义后，理想的行为是：“我抱歉地通知您，由于当前数据源接口受限，我无法获取到关于 XXX 的最新实时数据。基于我截止到 2026 年的离线知识，该情况是……（给出备用方案），建议您稍后再试。”
- 这个问题如何泛化到"所有工具失败的场景"？（不只是限流，还有超时、404、API down 等）
- 泛化到所有工具失败：我建议建立统一的工具输出包装器（Wrapper）。无论是超时、404 还是凭证过期，统一转化成 LLM 能听懂的业务限制描述，而不是把 Python 的 Traceback 报错直接丢给 LLM。
