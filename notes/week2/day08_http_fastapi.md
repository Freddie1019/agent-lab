# Day 8 概念卡片：HTTP 与 FastAPI

---

## 卡片 1：HTTP 协议的本质 🤖

HTTP 是一个**应用层、无状态、请求-响应**的通信协议。

核心特征：
- **无状态**：每个请求独立，服务器不"记得"上次（这是为什么需要 cookie、session、JWT 等机制）
- **基于文本**（HTTP/1.x）：报文是人类可读字符串，易调试
- **应用层协议**：底层基于 TCP，保证可靠传输
- **同步阻塞模型**：客户端发请求等响应（这是 LLM 调用慢导致 Agent 必须异步的根源）

一次 HTTP 通信 = 一对请求/响应报文，各包含三部分：
- 起始行（请求行 / 状态行）
- 头部（Headers，键值对）
- 主体（Body，可选）

---

## 卡片 2：HTTP 方法的语义 🤖

5 个核心方法，背后是"资源操作"的抽象：

| 方法 | 语义 | 幂等 | Agent 场景 |
|------|------|-----|-----------|
| GET | 读取 | ✅ | 查询任务状态 |
| POST | 创建/执行 | ❌ | 启动研究任务 |
| PUT | 全量替换 | ✅ | 更新整个配置 |
| PATCH | 部分更新 | ⚠️ | 改个字段 |
| DELETE | 删除 | ✅ | 取消任务 |

**幂等性**：同样的请求执行 1 次和 N 次效果一样。它决定了"重试是否安全"——重试 GET 没事，重试 POST 可能创建多次。

**反模式**：用 GET 做修改性操作（浏览器/代理会缓存或预取，导致动作意外执行多次）。

## 卡片 3：HTTP 状态码的工程意义 🤖

3 位数字，第一位决定大类：

- **2xx 成功**：200 OK、201 Created（POST 后）、204 No Content（DELETE 后）
- **4xx 客户端错误**：你的请求有问题
  - 400 Bad Request（格式错）
  - 401 Unauthorized（没认证）
  - 403 Forbidden（认证了但没权限）
  - 404 Not Found
  - 422 Unprocessable Entity（参数语义错）
  - 429 Too Many Requests（限流）
- **5xx 服务器错误**：我的代码有 bug
  - 500 Internal Server Error
  - 503 Service Unavailable（下游挂了）

**设计原则**：
- 4xx vs 5xx 的本质是"谁的锅" —— 客户端错误别返回 5xx
- 401 vs 403：401 = "你是谁？" / 403 = "我知道你是谁但你没权限"

HTTP 状态码 = 网络层的错误分类。你 Day 6 设计的 `AgentError` 体系 = 应用层的错误分类。两者一脉相承。

## 卡片 4：FastAPI / uvicorn / ASGI 三件套 🤖

很多人搞混它们的关系：

- **FastAPI** 是**框架**——定义"路由 → Python 函数"的映射规则、参数校验、自动文档
- **uvicorn** 是**服务器**——真正监听端口、接收 HTTP 报文、调度 FastAPI
- **ASGI** 是**接口标准**——FastAPI 和 uvicorn 之间的对话协议（异步版的 WSGI）

- 浏览器 ─HTTP─► uvicorn ─ASGI─► FastAPI ─调用─► 你的函数

类比：FastAPI 是菜谱，uvicorn 是厨师，ASGI 是它们之间的语言。

**为什么 AI 项目首选 FastAPI**：
- 原生异步（LLM 调用是 IO 密集，必须 async）
- Pydantic 集成（数据校验对 Agent 复杂参数特别重要）
- 自动生成 OpenAPI 文档（接口即文档）
- 类型注解驱动（不是装饰，是核心机制）

## 卡片 6：类型注解 = 服务契约 ✍️

FastAPI 用 Python 类型注解做参数校验。请回答：

- 这种"类型即文档、类型即校验"的设计相比"传统手写校验代码"有什么优势和潜在风险？
- 优势：消除了大量的 if not isinstance(x, int): return 400 这类面条代码（Boilerplate）。代码即单一事实源（Single Source of Truth），改了类型注解，API 校验和 Swagger 文档自动同步更新。
- 风险（性能与灵活性隐患）：Pydantic 的运行时校验需要消耗 CPU。如果传输超大 JSON 列表，反序列化和校验会变成性能瓶颈。同时，过于严格的类型会导致弱类型前端（或某些多变的数据源）因微小的格式错位而被一刀切地拦截。

- 你今天故意传错类型时（`max_steps="abc"`），FastAPI 返回了什么？这对前端工程师有什么意义？
- 当传错类型时，FastAPI返回的是 422 Unprocessable Entity状态码，并且附带了一个及其结构化的Body（包含那个字段错、期望什么类型、实际传了什么）
- 对前端的意义：不用再去猜测为什么报错了，可以直接去解析错误JSON获取准确错误信息
- 你之前 AI 调 LLM 时也用过 schema（Day 2 的 tool schema），两者有什么共同点？
- 两者本质上都是 JSON Schema 的实例化应用。Day 2 的 Schema 是你写给 LLM 看的契约（限制大模型的参数生成）；
- 而 FastAPI 的 Pydantic 模型是你写给人类/外部客户端看的契约。不管是 AI 还是人类，要在工程中安全通信，必须基于结构化的契约。

## 卡片 7：把函数变成 HTTP 接口的本质 ✍️

任务 4 你把 `ResearchAgent.run()` 包成了 HTTP 接口。请回答：

- 从"调用本地函数"到"通过 HTTP 调用"，本质上多了哪些工程问题？
- 从调用本地函数到HTTP接口，必须开始处理：网络不可靠（丢包、超时）、序列化损耗（对象转JSON文本）、多线程/多进程并发竞争、接口鉴权与安全防护、数据持久化隔离等本地调用根本遇不到的问题
- 你的 `/research` 接口目前是**同步阻塞**的——这会带来什么问题？什么场景下会暴露？
- 问题：Agent任务通常耗时极长，如果是同步阻塞的，服务器的worker线程会被这个请求死死卡住

- 提示：想想"100 个用户同时请求研究任务"会发生什么

开放题：
你今天写了 POST /research 接口，它是同步阻塞的——客户端发请求后，HTTP 连接挂在那等 30 秒（甚至 5 分钟）直到 Agent 完成才拿到响应。
这种设计在生产环境会出现哪些严重问题？至少列 3 个。
然后设想一下：如果让你设计"长任务接口"，你会怎么改进？（提示：从用户体验、服务器资源、可靠性等角度想）
三五句话亮观点，我们辩两轮。这道题正好为 Day 10 流式输出和 Week 4 任务队列做铺垫。


针对长任务接口的改进，我想从最具生产实战价值的方案“异步任务队列+轮询”架构，搭配SSE进行流式进度推送
具体设计：
解耦长任务：用户发起 POST /research 时，服务端不等待 Agent 执行，而是立即向消息队列（如 Celery/Redis）派发任务，
并秒级返回 202 Accepted 以及一个 task_id。这样可以瞬间释放 Uvicorn 线程，彻底解决线程池耗尽的雪崩隐患。多通道获取结果：
后端兜底：前端可以通过 GET /tasks/{task_id} 定时轮询任务状态（PENDING / RUNNING / SUCCESS）。
动态体验提升：利用 SSE（服务器发送事件） 建立轻量级长连接，由服务端主动向前端实时推送 Agent 的思考状态和工具调用进度（例如：“正在搜索...”、“正在抓取网页...”）。
这种“后台异步消费 + 前端无感流式交互”的组合，既保证了 Web 服务器的高并发稳定性，又给予了用户极其丝滑、不白屏的动态交互体验。这也是 Week 2 后续走向工业级 Agent 服务的标准解法！
