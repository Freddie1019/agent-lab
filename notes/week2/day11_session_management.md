# Day 11 概念卡片：会话管理与多用户隔离

---

## 卡片 1：HTTP 无状态 vs 用户有状态 🤖

HTTP 协议本身无状态，但用户体验天然需要有状态（"它指代上文"）。

三种方案：
- 客户端持有状态：简单，但跨设备丢失、安全风险
- 服务端持有状态：体验好，但服务器复杂
- 混合方案：服务端权威 + 客户端缓存

LLM 应用必须用服务端方案，因为历史长、有数据价值、涉及隐私。
ChatGPT / Claude.ai 全部如此。

## 卡片 2：会话数据模型的三层 🤖
User (用户)

└─ Session (一次对话)

└─ Message (一条消息)

关键决策：
- Session 粒度：ChatGPT 风格（用户主动开启）vs 微信风格（无限延续）
- Agent 通常用 ChatGPT 风格——允许用户开启新对话避免污染

Message 字段对应 LLM API 的 messages 数组——本质是**持久化你 Day 1 的内存数组**。

## 卡片 3：多用户隔离的安全边界 🤖

三大陷阱：
1. **IDOR 漏洞**：直接用 session_id 取数据，忘记检查 user_id 所有权
2. **缓存污染**：缓存不考虑 user_id，跨用户数据泄露
3. **日志泄露**：把对话内容打日志，运维能看见

业界标准：
- 每次访问 session 都做 `assert_ownership()` 校验
- **返回 404 而不是 403**（防止枚举攻击）

工程层硬约束，不依赖客户端自觉。

---

## 卡片 4：会话级互斥锁防并发污染 🤖

同一 session 被并发修改时：
- 两个请求读到同一个旧快照
- 各自跑完追加消息
- 第二个请求基于过期上下文，回答可能不连贯

业界解法：
- 方案 1：会话级互斥锁（asyncio.Lock）—— 我们今天做的
- 方案 2：拒绝并发（返回 409 Conflict）
- 方案 3：分支会话（ChatGPT 用的，让用户显式分叉）

**锁必须是会话级的，不能全局一把锁——否则不同用户互相阻塞。**

## 卡片 5：会话状态的存储介质权衡 ✍️

请回答：

- 你今天用了内存字典作为会话仓库。它的 3 个致命缺陷分别是什么？每个会在什么场景暴露？
- 1. 单点故障和数据缺失：一旦进程崩溃、服务器断电等情况发生，内存中的数据就会瞬间蒸发
- 2. 无法扩展：当用户量增加时，需要部署多台服务器，但是内存却无法跨机器共享，用户的请求打到机器A有历史，打到机器B就会报会话不存在
- 3: 内存溢出：历史通常包含大量上下文、网页抓取结果，随着会话积累，内存会无脑膨胀，最终撑爆服务器进程

- Redis、PostgreSQL、内存这三个存储介质，各适合存"会话的哪一部分数据"？
- 内存： 仅用于单次请求生命周期内的会话或超高频的本地局部变量
- Redis：Session元数据（当前状态、最近3轮滑窗历史、活跃心跳、临时摘要）
- PostgreSQL：全量历史消息 （Message 归档表）、审计日志（Audit Log）、用户画像、结构化事实库。
- 提示：考虑读写频率、持久化要求、查询模式

## 卡片 6：会话历史与 Day 4 上下文管理的关系 ✍️

请基于今天的实战回答：

- 用户能看到的"完整历史"和喂给 LLM 的"压缩历史"为什么必须分开存？
- 1.功能定位不同：“完整历史”是给人（前端）看的。用户需要看到自己说过的每一句原话、错别字，以及 Agent 当时最原始的回答。“压缩历史”是给 LLM（大脑）吃的。
- 因为 Context Window 的成本和 Lost in the Middle 限制，我们必须对原始文本进行裁剪、切片或摘要，过滤掉多余的工具返回噪音，只保留高密度的信息。

- 如果我用摘要式压缩（Day 4 的方案），多轮对话会不会触发"递归崩溃"？（你 Day 4 实证发现的那个）
- 基础轨（完整全量存储）：数据库中永远雷打不动地保存 100% 真实的原始 Message 数组。
- 动态生成轨（滑窗 + Pinning）：每次把数据喂给 LLM 前，临时、动态地从 DB 里捞出：
- [System Prompt] + [固定的核心用户信息/Pinning数据] + [从全量历史中截取的最近 N 轮原始消息]。如果确实需要摘要，永远基于最- - 底层的原始消息去生成全局的一手摘要，坚决不对摘要进行“二次迭代压缩”。
- 怎么避免？提示：滑窗 + 关键事实 pinning 的组合
